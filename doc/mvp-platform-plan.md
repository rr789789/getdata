# 最小 IoT 平台实施计划

## 1. 目标

本项目按“类似涂鸦 / ThingBoard 的最小可行平台”理解，先实现一个可直接运行的单二进制版本，满足以下要求：

- 支持 Windows / Linux 直接编译为二进制运行
- 支持设备接入、鉴权、心跳、遥测上报、服务端下发命令
- 提供基础管理 API
- 代码结构可扩展，后续可演进到 200k 设备接入
- 优先保证可靠性、可观测性、并发安全和可维护性

说明：

- 用户描述中的 “mpv” 这里按 “MVP” 处理
- 第一阶段先做单节点最小平台，不在首版实现完整多租户、规则引擎、可视化大屏、复杂权限体系

## 2. MVP 范围

首版必须具备：

- 设备注册
- 设备凭证管理
- 设备长连接接入
- 遥测数据接收
- 命令下发
- 命令回执
- 健康检查
- 基础运行指标

首版暂不实现：

- 多租户
- MQTT 全协议兼容
- OTA
- 可视化仪表盘
- 规则引擎
- 时序数据库
- 集群调度

## 3. 技术选型

### 3.1 语言

使用 Go 实现，原因：

- 原生支持高并发网络模型
- 跨平台交叉编译成熟
- 能输出单文件二进制，适合 Windows / Linux 直接运行
- 标准库网络栈稳定，依赖可控

### 3.2 运行形态

首版使用单进程单二进制，内部模块化：

- HTTP 管理接口
- TCP 设备接入网关
- 设备会话管理
- 命令路由
- 遥测存储
- 运行指标与健康检查

### 3.3 协议策略

首版设备接入协议采用简单的 `TCP + JSON Lines`：

- 每条消息一行 JSON
- 设备先发送 `auth`
- 之后发送 `ping` / `telemetry` / `ack`
- 平台可主动下发 `command`

原因：

- 开发速度快，便于打通端到端链路
- 比 HTTP 轮询更接近真实设备长连接场景
- 后续可以在不破坏核心服务层的前提下增加 MQTT / WebSocket 适配器

## 4. 架构设计

### 4.1 首版逻辑分层

1. `API 层`
   - 提供设备注册、查询、命令下发接口
2. `Gateway 层`
   - 处理设备 TCP 连接、鉴权、消息解码、回写
3. `Core 层`
   - 管理设备在线状态、命令路由、遥测入库
4. `Store 层`
   - 抽象设备元数据和遥测存储

### 4.2 关键对象

- `Device`
  - `id`
  - `name`
  - `token`
  - `metadata`
  - `created_at`
- `Session`
  - `device_id`
  - `conn_id`
  - `connected_at`
  - `last_seen`
- `Telemetry`
  - `device_id`
  - `ts`
  - `values`
- `Command`
  - `id`
  - `device_id`
  - `name`
  - `params`
  - `status`
  - `created_at`

## 5. 可靠性设计

首版就要落实的可靠性约束：

- 所有共享状态必须并发安全
- 所有网络写入使用超时与有界队列，避免慢连接拖垮系统
- 所有 goroutine 受 `context` 生命周期管理
- 提供优雅退出，停止接收新连接并清理资源
- 所有输入做基本校验，避免非法包放大风险
- 结构化日志，便于定位问题
- 存储层使用接口隔离，便于后续替换为持久化实现

## 6. 200k 设备接入的演进路线

200k 设备不是首版单机硬指标，而是设计目标。为此首版必须保证后续可平滑拆分。

### 6.1 首版到中期的演进路径

第一阶段：单节点单二进制

- 适合开发联调和功能验证
- 默认使用内存存储

第二阶段：单节点增强

- 设备元数据切换到持久化 KV / SQL
- 遥测写入消息队列或日志流
- 增加 Prometheus 指标
- 增加连接限流和背压控制

第三阶段：多节点集群

- Gateway 与 Core 拆分
- 会话状态放入共享注册中心
- 命令通过消息总线路由
- 遥测进入 Kafka / Pulsar / NATS JetStream
- 元数据用 PostgreSQL / MySQL
- 热点数据走 Redis

### 6.2 200k 设备建议架构

- `gateway`：无状态接入层，多实例水平扩容
- `session-router`：维护设备在线路由
- `core-service`：命令、业务编排、权限控制
- `telemetry-ingest`：批量写入流系统
- `storage`：元数据 + 时序/日志存储
- `metrics/alerting`：Prometheus + Grafana + Alertmanager

### 6.3 容量关键点

- 每连接一个 goroutine 可行，但必须控制每连接缓冲区大小
- 命令下发必须异步化，避免 API 线程阻塞
- 遥测写入必须支持削峰填谷，不能直接同步写数据库
- 心跳和空闲连接清理必须自动化
- 必须预留协议适配层，后续接 MQTT 时不能重写核心逻辑

## 8. 第一轮开发内容

本轮代码实现目标：

1. 初始化 Go 项目
2. 提供基础配置加载
3. 实现内存版设备存储与遥测存储
4. 实现 HTTP 管理 API
5. 实现 TCP 设备接入网关
6. 打通设备鉴权、遥测、命令、回执闭环
7. 提供健康检查和基础指标接口
8. 配置 GitHub Actions，输出 Windows / Linux 二进制

## 9. 第一轮验收标准

- `go test ./...` 可通过
- 本地可直接启动单二进制服务
- 可通过 HTTP 注册设备
- 设备可通过 TCP 认证接入
- 设备可上报遥测
- 平台可向在线设备发送命令
- Windows / Linux 能通过 GitHub Actions 编译出可执行文件
